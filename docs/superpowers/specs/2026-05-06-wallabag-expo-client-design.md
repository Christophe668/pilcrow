# Wallabag Expo Client — Design

**Date:** 2026-05-06
**Status:** Approved (brainstorming)
**Reference prototype:** `wallabag-prototype.html` at repo root

## 1. Goal

A general-purpose, cross-platform Wallabag client written once with Expo, shipping for iOS, Android, and web. Users connect to any Wallabag instance by supplying a server URL and OAuth2 password-grant credentials. The visual language follows the prototype: warm paper aesthetic, serif display, oklch-derived palette, three-pane desktop layout collapsing responsively to phone.

The MVP includes:

- List articles with filters (Unread, Starred, Archive, All)
- Read articles with full HTML rendering
- Toggle starred / archived / unread, delete
- Add article by URL
- Tags: view, filter, add/remove
- Full-text search
- Annotations / highlights
- Offline reading (articles cached locally)
- Share-to-app: iOS share extension, Android intent filter, web bookmarklet

Out of scope for MVP: multi-account, background sync while app is closed, push notifications, iPad multi-window, wearables, any server-side change.

## 2. Tech stack

| Concern | Choice |
|---|---|
| Language | TypeScript |
| Framework | Expo (prebuild flow, not managed) |
| Navigation | Expo Router (file-based) |
| Data fetching / cache | TanStack Query |
| Local DB / offline | Expo SQLite (WAL mode) |
| Secret storage | Expo SecureStore + native Keychain access group (iOS) |
| Non-secret storage | AsyncStorage |
| Styling | NativeWind (Tailwind for RN) |
| Forms | React Hook Form + Zod |
| HTML rendering | `react-native-webview` (mobile) / sandboxed `<iframe srcdoc>` (web) |
| Package manager | pnpm |
| Errors / crashes | Sentry (opt-in) |

Rationale: NativeWind unifies styling with the prototype's utility-style CSS and works on web. SQLite-first lets the UI ignore the network. Prebuild is required because the iOS share extension needs a real native target.

## 4. Architecture overview

```
              ┌────────────────────────┐
              │ Expo Router screens    │
              │ (app/**, src/components)│
              └─────────┬──────────────┘
                        │ TanStack Query hooks
                        ▼
              ┌────────────────────────┐
              │ Repositories           │
              │ (src/db/repos/*)       │
              └─────────┬──────────────┘
                        │
                        ▼
              ┌────────────────────────┐
              │ SQLite (single source  │
              │ of truth for the UI)   │
              └─────────┬──────────────┘
                        ▲
                        │
              ┌─────────┴──────────────┐
              │ Sync engine (src/sync) │ ←→ Wallabag REST (src/api)
              └────────────────────────┘
```

The UI never branches on online/offline. It reads from SQLite. The sync engine handles network I/O and writes results into SQLite. Mutations apply optimistically to SQLite and enqueue payloads in the outbox; the drainer flushes the outbox when online.

## 5. Auth & onboarding

### 5.1 Two-step wizard

**Step 1 — Server URL.** Single input. On Continue:

```
GET {url}/api/info.json
```

200 with a `wallabag` field validates the URL (and gives us the version, useful for capability flags later). Store normalized URL in AsyncStorage. Advance.

**Step 2 — Credentials.** Inputs: client_id, client_secret, username, password. A help link "Need a client_id/secret?" deep-links to `{url}/developer/client/create` to make the friction visible. On Sign in:

```
POST {url}/oauth/v2/token
  grant_type=password
  client_id, client_secret, username, password
→ { access_token, refresh_token, expires_in }
```

Persist `client_id`, `client_secret`, `username`, `access_token`, `refresh_token`, and `token_expires_at` (computed `now + expires_in`) into SecureStore. Discard the password — it's never stored.

### 5.2 Token refresh

`src/api/client.ts` exposes a single `request()` wrapper. Before each call:

1. If `token_expires_at` is within 60s of now, refresh first.
2. Otherwise, send the request.
3. If a response is 401 with `error: invalid_grant`, refresh once and retry.

Refresh:

```
POST {url}/oauth/v2/token
  grant_type=refresh_token
  refresh_token, client_id, client_secret
```

A single in-flight refresh promise serializes concurrent requests. On refresh failure, wipe SecureStore and redirect to `/(auth)/credentials` with the server URL prefilled.

### 5.3 Sign out

Clear SecureStore, AsyncStorage, drop the SQLite database file, navigate to `/(auth)/server`.

## 6. Data layer

### 6.1 SQLite schema

```sql
CREATE TABLE articles (
  id INTEGER PRIMARY KEY,            -- Wallabag entry id (negative for pending creates)
  title TEXT,
  url TEXT NOT NULL,
  domain_name TEXT,
  content TEXT,                      -- sanitized HTML from server
  preview_picture TEXT,
  reading_time INTEGER,
  language TEXT,
  is_archived INTEGER NOT NULL DEFAULT 0,
  is_starred INTEGER NOT NULL DEFAULT 0,
  created_at TEXT,
  updated_at TEXT,
  starred_at TEXT,
  archived_at TEXT,
  published_at TEXT,
  published_by TEXT,
  scroll_position REAL DEFAULT 0,    -- 0..1, drives resume-on-reopen
  server_updated_at TEXT,
  local_updated_at TEXT,
  pending_op TEXT                    -- 'create' | 'update' | 'delete' | NULL
);

CREATE TABLE tags (
  id INTEGER PRIMARY KEY,
  label TEXT NOT NULL,
  slug TEXT NOT NULL
);

CREATE TABLE article_tags (
  article_id INTEGER NOT NULL,
  tag_id INTEGER NOT NULL,
  PRIMARY KEY (article_id, tag_id)
);

CREATE TABLE annotations (
  id INTEGER PRIMARY KEY,
  article_id INTEGER NOT NULL,
  quote TEXT NOT NULL,
  ranges_json TEXT NOT NULL,         -- Wallabag-format range array
  text TEXT,                         -- optional note
  created_at TEXT,
  updated_at TEXT,
  pending_op TEXT
);

CREATE TABLE images (
  article_id INTEGER NOT NULL,
  src TEXT NOT NULL,
  local_path TEXT,
  status TEXT NOT NULL,              -- 'pending' | 'cached' | 'failed'
  size_bytes INTEGER,
  cached_at TEXT,
  PRIMARY KEY (article_id, src)
);

CREATE TABLE outbox (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  op TEXT NOT NULL,                  -- 'createEntry' | 'updateEntry' | 'deleteEntry' | 'addTag' | 'removeTag' | 'createAnnotation' | 'updateAnnotation' | 'deleteAnnotation'
  payload_json TEXT NOT NULL,
  created_at TEXT NOT NULL,
  attempts INTEGER NOT NULL DEFAULT 0,
  last_error TEXT
);

CREATE TABLE sync_state (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);

CREATE VIRTUAL TABLE articles_fts USING fts5(
  title, content, url, tags,
  content='articles', content_rowid='id'
);
```

FTS5 triggers keep the virtual table in sync with `articles`.

### 6.2 Sync engine

Three entry points, all idempotent:

- `runInitialSync()` — full metadata pull. Paginate `GET /api/entries.json?detail=metadata&perPage=100` until exhausted. Content is loaded lazily on first article open.
- `runIncrementalSync()` — `GET /api/entries.json?since={last_sync}&detail=full&perPage=100`. Update `sync_state.last_sync_since` from the max `updated_at` in the response.
- `drainOutbox()` — process outbox FIFO, exponential backoff per row, capped at 5 retries before surfacing the row as a "Failed save" in Settings.

Triggers:

- App launch (after auth gate)
- App foreground
- Pull-to-refresh on a list
- Every 5 minutes while in foreground (timer)
- After every successful mutation (drains outbox immediately)

Conflict resolution: last-write-wins by `server_updated_at`. If we have a pending mutation and the server side has changed since `local_updated_at`, the server wins for the conflicting field but our pending mutation still applies (we re-POST it).

### 6.3 Outbox semantics

Every mutation:

1. Repo writes the new state into SQLite + sets `pending_op` and bumps `local_updated_at`.
2. Repo enqueues a row in `outbox` with the API payload.
3. Drainer picks up rows in order, calls the API, on success clears `pending_op` and deletes the outbox row.

Order matters for "create then update": negative-id rows in `outbox` carry a temp id. On create success we update both the article row and any subsequent outbox payloads to reference the real id.

### 6.4 Image caching

When the reader prepares HTML, every `<img src>` is rewritten:

- If `images` row has `status='cached'` → use `file://{local_path}`
- Otherwise → use the network URL, and `INSERT OR IGNORE` a row with `status='pending'`; a background worker downloads via `expo-file-system`.

LRU eviction at a configurable cap (default 500 MB), shown in Settings → Sync.

### 6.5 TanStack Query keys

```
["articles", { filter: "unread" | "starred" | "archive" | "all" }]
["articles", { tag: string }]
["articles", { search: string }]
["article", id]
["tags"]
["annotations", articleId]
["sync-status"]
```

Queries read from SQLite. Mutations call repos. `queryClient.invalidateQueries` is driven by repo write events emitted on a small bus.

## 7. Screens & navigation

### 7.1 Responsive layout

| Breakpoint | Layout |
|---|---|
| ≥1280px | Three-pane: rail (240px) · list · meta (280px) |
| ≥768px | Two-pane: icon rail · list+reader; meta becomes slide-over |
| <768px | Single column: bottom tab bar (Library, Add, Search, Settings); list pushes reader |

NativeWind handles breakpoints with `md:` / `lg:` / `xl:` prefixes. Components are shared; only the layout shell differs.

### 7.2 Routes (Expo Router)

```
/(auth)/server                  Step 1
/(auth)/credentials             Step 2

/(app)/(library)/index          Unread (default)
/(app)/(library)/starred
/(app)/(library)/archive
/(app)/(library)/all
/(app)/(library)/tags/[tag]
/(app)/(library)/search?q=...

/(app)/article/[id]
/(app)/add                      Modal presentation
/(app)/settings
```

Desktop: opening an article from the list keeps rail+list visible and renders the article in the right pane via parallel routing. Phone: opening pushes a new stack screen.

### 7.3 Library shell

- **Rail.** Sections: Library (Unread/Starred/Archive/All), Tags (top N + "All tags" sheet), Settings shortcut. Active route highlighted via `useSegments()`.
- **List header.** Display title (serif 34px, matching prototype `.list-title`), count, sort menu (newest / oldest / longest / shortest), filter chips for active tag/search.
- **List.** Virtualized with `@shopify/flash-list`. Row: title, domain · reading-time · age, optional preview image, tag chips, star/archive icons. Pull-to-refresh → `runIncrementalSync()`.
- **Meta pane (desktop only).** Hero image, title, byline, action row, tag chips, "Open reader" button.
- **Empty states.** Per filter, illustrated.

### 7.4 Search

Local first via SQLite FTS5. Instant results as you type. On submit, if local hits < 5, also fetch `GET /api/search.json?term={q}` and merge results not already present locally.

### 7.5 Settings

- Account: server URL (read-only), username, sign out.
- Sync: last sync time, "Sync now", cache size, "Clear cache", failed saves list.
- Reader: font size, font family, line height, theme, width.
- About: app version, server version (from `/api/info.json`), Wallabag docs link.
- Diagnostics (advanced): last 200 sync events, last 20 API errors, outbox state, copy-to-clipboard.

## 8. Reader

### 8.1 Rendering pipeline

1. SQLite gives `article.content` (already-sanitized HTML from the server).
2. Rewrite `<img src>` against the `images` table.
3. Inject a `<style>` block: prototype typography rules + user reader prefs.
4. Inject `reader-bridge.js`, which:
   - Reports scroll position to native (debounced) for resume.
   - Listens for selection events, shows a custom selection menu (Highlight / Note / Copy / Look up).
   - Renders existing annotations as `<mark data-annotation-id="...">` wrappers.
   - Posts annotation create/update events to native via `postMessage`.
5. Native chrome around the WebView: top bar, bottom action bar, top progress indicator.

On web, the same pipeline drives a sandboxed `<iframe srcdoc>` with `allow-same-origin allow-scripts`. `window.postMessage` carries the bridge protocol.

### 8.2 Annotations

Wallabag stores annotations with a `ranges` array of XPath-like selection descriptors (`{ start, startOffset, end, endOffset }`).

- **Create.** Capture `Selection.getRangeAt(0)` → serialize to Wallabag's range format → optimistic insert into `annotations` + outbox → wrap selection with `<mark>`.
- **Render.** On article open, load annotations and re-apply `<mark>` wrappers via a TreeWalker.
- **Tap a mark.** Bottom sheet with note text + edit/delete.
- **Edit note.** Updates `annotations.text` + outbox `updateAnnotation`.

Range serialization lives in `src/lib/range-serializer.ts` and is heavily unit-tested. We use a small library (`rangy` or hand-rolled) wrapped behind our own interface so we can swap implementations.

### 8.3 Reader prefs

Stored in AsyncStorage. Applied by re-injecting the `<style>` block.

| Pref | Values |
|---|---|
| Font family | Newsreader (default) / system-sans |
| Size | S, M, L, XL |
| Line height | 1.4, 1.6, 1.8 |
| Theme | auto, light, dark, sepia |
| Width | comfortable (680px), wide (820px) |

### 8.4 Action bar

Star, Archive (toggles to Unarchive), Add tag (sheet), Share (system share sheet), More (delete with confirm, view original in browser, copy link). All optimistic, all routed through repos+outbox.

### 8.5 Resume

`scrollY / scrollHeight` reported by the bridge on a 250ms debounce; native writes to `articles.scroll_position`. On reopen, scroll restores after first paint.

## 9. Add article + share targets

### 9.1 Add modal (in-app)

Route `/(app)/add`. Inputs: URL (prefilled from clipboard if it contains a URL), optional tags. Submit:

1. Optimistic insert with temp negative id, `pending_op='create'`, URL as title placeholder, `loading` flag.
2. Outbox push: `POST /api/entries.json` with `{ url, tags }`.
3. On success, rewrite the row with the real id + server-fetched metadata; clear `loading`.
4. On failure, retain the row with an error badge; user can retry from Settings → Failed saves.

### 9.2 iOS share extension

Custom Expo config plugin `plugins/with-ios-share-extension.ts` adds a `WallabagShareExtension` target during prebuild.

- Type: Action/Share Extension.
- `NSExtensionActivationRule`: URL or web page.
- UI: minimal SwiftUI view — title (prefilled from shared content), tags input, Save / Cancel.
- App Group `group.com.cpecsteen.wallabag` shared with the main app.
- Keychain access group set up via `plugins/with-keychain-access-group.ts` so the extension can read the access_token written by the main app.
- On Save: extension calls `POST /api/entries.json` directly. On failure it writes the pending payload to a JSON file inside the App Group; the main app drains it into the outbox on next launch.
- Token refresh inside the extension: same logic, single attempt only (extensions have ~30s budget).

### 9.3 Android share intent

`plugins/with-android-share-intent.ts` adds an `<intent-filter>` to `MainActivity` for `android.intent.action.SEND` with `text/plain`.

Two modes (Settings toggle):

- **Quick save (default):** consume the intent in `MainActivity.onNewIntent`, save silently in the background via the JS bridge, show a Snackbar/toast.
- **Open add modal:** route the intent into `/(app)/add` prefilled.

If the shared text isn't a URL, fall back to "create entry from text" (Wallabag doesn't support this directly → surface a toast: "Wallabag needs a URL to save").

### 9.4 Web bookmarklet

Settings page exposes:

```
javascript:(()=>{location.href='https://app.example.com/add?url='+encodeURIComponent(location.href)})()
```

Drag-to-bookmarks. Saves whatever page you're on into Wallabag via the web app.

## 10. Theming

- **Tokens** in `src/theme/tokens.ts`. The prototype's oklch values are precomputed to hex/rgb at build time by `scripts/oklch-to-hex.ts`. Token names match the prototype (`bg`, `surface`, `surface-2`, `fg`, `muted`, `subtle`, `border`, `border-strong`, `accent`, `accent-ink`, `accent-soft`, `teal`, `highlight`).
- **Tailwind config** consumes those tokens, so NativeWind classes like `bg-surface-2 text-fg border-border` work everywhere.
- **Modes:** light / dark / sepia / auto. User override in Settings. `Appearance` listener flips at runtime.
- **Fonts:** Newsreader via `expo-font` (variable font, three weights). System sans for body.
- **Web typography:** the same tokens drive a `<style>` element injected into the document `<head>`.

## 13. Observability

- Sentry on mobile + web. Off by default in dev. Opt-in via env.
- Local debug log viewer in Settings → Diagnostics: last 200 sync events, last 20 API errors, outbox state. Long-press to copy. No external telemetry beyond Sentry.

## 14. Security & privacy

- Tokens live only in SecureStore (Keychain on iOS, Keystore on Android). Web uses an HttpOnly-ish equivalent: tokens go into `localStorage` (acceptable for a self-hosted client; documented).
- The password is never stored — discarded after the initial token exchange.
- Sign out drops SecureStore + AsyncStorage + the SQLite file.
- The iOS extension reads tokens via a Keychain access group; no token ever transits the App Group file (file only carries pending mutations).
- All HTTP requests to the configured Wallabag instance only. No third-party network calls except Sentry (opt-in) and font CDN (eliminated by bundling Newsreader locally).

## 15. Out of scope (MVP)

- Multi-account
- Background sync while app is fully closed (`expo-background-fetch` deferred)
- Push notifications
- iPad multi-window UX
- Wearables
- Server-side changes to Wallabag

## 16. Open decisions deferred to implementation

- Final iOS/Android bundle id and App Group id (working default: `com.cpecsteen.wallabag` / `group.com.cpecsteen.wallabag`; subject to change before first store submission)
- Production web origin used by the bookmarklet (placeholder: `https://app.example.com`)
- Web hosting target (Cloudflare Pages vs Vercel)
- Whether to use `rangy` or a hand-rolled range serializer (decide during the implementation of the reader)
- Exact image-cache LRU policy parameters (start with 500 MB, tune)
